## 索引是一个有代价的独立数据结构

在讨论具体索引类型之前，先搞清楚索引是什么。索引是 PostgreSQL 在主表旁边单独维护的一个磁盘数据结构。它用写入开销和存储空间换取特定查询模式下更快的读取。

每次 INSERT、UPDATE 和 DELETE 都必须更新所有相关索引。如果一张表有六个索引，每次写入就会触碰六个数据结构。理解这个权衡是所有索引决策的基础：当读取收益超过写入代价时加索引，删掉没有任何查询用到的索引。

## 复合索引有主导左列

`(a, b, c)` 上的复合索引不等于三个独立索引。它是一棵 B 树，先按 `a` 排序，在每个 `a` 值内再按 `b` 排序，在每个 `(a, b)` 对内再按 `c` 排序。

这个结构意味着这个索引对只过滤 `a`、或 `(a, b)`、或 `(a, b, c)` 的查询有用。对只过滤 `b` 或只过滤 `c` 的查询通常没用，因为树不是按这些列独立排序的。

结论：设计复合索引时，如果查询会单独过滤某列，把选择性最高的列放最前面。把等值过滤放在范围过滤之前，因为范围过滤会终止后续列的有序遍历。

## 部分索引只覆盖有用的子集

部分索引只包含满足某个条件的行：

```sql
CREATE INDEX orders_pending_idx ON orders (created_at)
WHERE status = 'pending';
```

这个索引比全表索引更小、扫描更快，而且只对满足条件的行进行维护。过滤 `status = 'pending'` 并在 `created_at` 上排序或过滤的查询可以直接用它。

部分索引被严重低估。任何时候一个查询总是过滤某个低基数列的特定值，都可以考虑在这个子集上建部分索引，而不是在全部值上建完整索引。

## EXPLAIN ANALYZE 告诉你实际发生了什么

读查询计划是值得培养的技能。`EXPLAIN ANALYZE` 执行查询并同时展示计划策略和实际执行统计：

```sql
EXPLAIN ANALYZE SELECT * FROM events WHERE user_id = 42 ORDER BY created_at DESC LIMIT 10;
```

重点关注：
- **Seq Scan vs. Index Scan**：大表上出现顺序扫描说明可能缺少索引
- **Rows** 估计 vs. 实际：差距很大说明统计信息过期了——运行 `ANALYZE`
- **Bitmap Heap Scan**：PostgreSQL 先收集索引命中，再按堆顺序取行；返回较多行时常见
- **Nested Loop vs. Hash Join**：嵌套循环适合内侧结果集小的情况；哈希连接更适合大数据集

计划器的代价估计是任意单位，但各节点之间的相对值告诉你时间花在了哪里。

## 仅索引扫描是最好的情况

当查询所需的所有列都能由索引满足而不需要访问堆时，PostgreSQL 执行仅索引扫描。这明显更快，因为完全避免了堆访问。

要启用仅索引扫描，索引必须包含查询投影的每一列。用 `INCLUDE` 添加非键列：

```sql
CREATE INDEX users_email_idx ON users (email) INCLUDE (name, created_at);
```

被包含的列存储在叶页面中但不参与排序。它们不能用于排序或范围过滤，但可以让索引在不访问堆的情况下满足投影需求。

对于高频查询且每次返回固定的少量列的场景，这个技巧价值最大。
